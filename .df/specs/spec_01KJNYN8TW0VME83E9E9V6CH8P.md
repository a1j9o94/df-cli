# Run output showcase: screenshots and logs for every completed run in the dashboard

## Goal

Every completed run should have a "What did this produce?" section in the dashboard. For visual specs (frontend, UI, dashboard): embedded screenshots of the running output. For CLI/backend specs: curated agent logs showing what happened. Runs become presentable artifacts for stakeholders without digging into code or terminals.

## Problem

Today a completed run shows: status badge, cost, scenario results. But it doesn't show what was actually built. A PM wants to see the UI that was created, or the key decisions an agent made, or a summary of what changed. Without this, the dashboard is an accounting tool (how much did it cost?) rather than a showcase (what did we ship?).

## Requirements

### Module 1: Output Tab in Dashboard

- New "Output" tab on each run's detail view, alongside Overview, Modules, and Validation
- The Output tab is the "show and tell" for the run
- If the run is still in progress, show a placeholder: "Build in progress. Output will appear when complete."
- If the run has no output data, show a clean empty state: "No output captured for this run." with a hint about how screenshots are captured

### Module 2: Screenshot Capture for Visual Runs

- Detection: a run is "visual" if the spec title or goal mentions: UI, frontend, dashboard, page, component, layout, design, screen, widget, or view (case-insensitive keyword match)
- Builder agents: during TDD cycles, if Playwright is available in the project, capture a screenshot after each passing test cycle. Store in `.df/runs/<run-id>/screenshots/build-<module>-<step>.png`
- Evaluator agent: after final validation of a visual spec, capture screenshots of the completed output. Store in `.df/runs/<run-id>/screenshots/eval-<scenario>.png`
- Screenshot metadata file: `.df/runs/<run-id>/screenshots/manifest.json`:
  ```json
  [
    {
      "filename": "build-data-layer-3.png",
      "phase": "build",
      "module": "data-layer",
      "step": 3,
      "caption": "After test cycle 3: table component renders with mock data",
      "timestamp": "2026-03-01T19:12:00Z"
    }
  ]
  ```
- If Playwright is not available, skip screenshot capture silently — do not fail the build
- Maximum 20 screenshots per run to avoid disk bloat. Keep the most recent per module plus all evaluator screenshots.

### Module 3: Screenshot Gallery in Dashboard

- The Output tab for visual runs shows a gallery grid of screenshots
- Each screenshot: thumbnail (click to expand full-size), caption, phase badge (Build / Eval), module name, timestamp
- Gallery is ordered chronologically — build screenshots first, evaluator screenshots last
- Expanded view: full-width image with navigation (prev/next)
- Screenshots served via `GET /api/runs/:id/screenshots/:filename` (reads from disk)
- Manifest served via `GET /api/runs/:id/screenshots` (returns the manifest JSON)

### Module 4: Curated Agent Log Highlights

For ALL runs (visual and non-visual):

- Extract significant events from agent logs into a curated highlights view
- Significant events (filtered from raw agent output):
  - Module creation: "Created module: <name> — <description>"
  - Test pass: "Scenario passed: <scenario-name>"
  - Test fail: "Scenario failed: <scenario-name> — <first line of error>"
  - Key decision: Lines matching patterns like "Decision:", "Tradeoff:", "Chose X because", "Architecture:"
  - Error/recovery: Agent errors and how they were resolved
  - Integration: "Module <A> integrated with <B>"
- This is NOT raw Claude output — it's a filtered, structured log
- Stored in `.df/runs/<run-id>/highlights.json`:
  ```json
  [
    {
      "type": "module_created",
      "module": "auth-middleware",
      "description": "JWT validation middleware for Express routes",
      "timestamp": "2026-03-01T19:05:00Z"
    },
    {
      "type": "scenario_passed",
      "scenario": "Login redirects to dashboard",
      "module": "auth-middleware",
      "timestamp": "2026-03-01T19:08:00Z"
    }
  ]
  ```
- Highlights are generated by the pipeline engine at the end of each phase (not in real-time during the build)

### Module 5: Module-by-Module Summary

- Within the Output tab, a "Modules" section shows a per-module summary card:
  - Module name and description
  - Files created/modified (list)
  - Tests passing (count)
  - Key decisions the agent made for this module (from highlights)
  - Build duration for this module
- This gives a PM-readable view of what each module contributed to the final output

### Module 6: CLI Export

- `dark run output <run-id>` — print the showcase data to stdout in a readable format:
  ```
  # Run Output: Fix authentication redirect loop

  ## Modules Built (3)

  ### auth-middleware
  - Files: src/middleware/auth.ts, src/middleware/auth.test.ts
  - Tests: 4/4 passing
  - Decisions: "Used JWT with refresh tokens instead of sessions for stateless auth"

  ### login-page
  - Files: src/pages/login.tsx, src/pages/login.test.tsx
  - Tests: 3/3 passing
  - Screenshots: 2 captured

  ...

  ## Scenarios: 8/8 passed
  ```
- `dark run output <run-id> --export` — write the output as a markdown file to `.df/runs/<run-id>/output.md` and print the path
- `dark run output <run-id> --export --format html` — export as a standalone HTML file with embedded screenshots (base64)

## Scenarios

### Functional

1. **Visual run showcase**: Build a spec that mentions "dashboard UI" in its goal. Verify the Output tab in the dashboard shows a screenshot gallery with build and evaluator screenshots.

2. **Backend run showcase**: Build a spec for an API endpoint (no visual keywords). Verify the Output tab shows curated log highlights and module summaries instead of a screenshot gallery.

3. **Evaluator screenshots captured**: Build a visual spec. After evaluation completes, verify screenshots exist in `.df/runs/<run-id>/screenshots/` with filenames starting with `eval-`.

4. **CLI export**: Complete a run. Run `dark run output <run-id> --export`. Verify a markdown file is written to `.df/runs/<run-id>/output.md` with module summaries and scenario results.

5. **Empty showcase graceful**: Complete a run where no screenshots were captured and agent logs have no highlight-worthy events. Verify the Output tab shows a clean message rather than an empty or broken view.

6. **Screenshot gallery navigation**: Complete a visual run with 5+ screenshots. Verify the gallery shows thumbnails, click to expand works, and prev/next navigation works.

7. **HTML export with embedded screenshots**: Complete a visual run. Run `dark run output <run-id> --export --format html`. Verify the HTML file is self-contained (screenshots base64-encoded) and renders correctly in a browser.

8. **Highlights filter noise**: Verify the highlights do NOT include raw Claude reasoning, token counts, or internal agent coordination messages — only the curated event types listed in Module 4.

### Changeability

1. **Add video capture**: Adding screen recording (in addition to screenshots) for visual runs should require adding a video capture step in the builder/evaluator agents and a video player in the gallery — no changes to the manifest format or API contract (add a `type` field to manifest entries).

2. **Add custom highlight patterns**: Allowing users to define custom patterns for log extraction (in `.df/config.yaml`) should require reading patterns from config in the highlight extractor — no changes to the dashboard rendering or storage format.
